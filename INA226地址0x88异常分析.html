🔍 INA226 地址异常分析

现象：原理图标注 地址：1000000（0x80 HAL 格式），但代码写 0x88 才能正常读到数据。原因并不是代码错误，也不是接线错误，而是原理图的硬件设计不符合 INA226 数据手册要求。

1. 现象描述

来源内容
原理图备注地址：1000000（= 7-bit 0x40 = HAL 0x80）
网表 A0/A1 连接A0 → 10kΩ → GND；A1 → 10kΩ → GND（应得 0x80）
实际代码INA226_Init(..., 0x88, ...)
实测结果代码写 0x88 能读到正确数据；写 0x80 读不到
⚠ 按原理图应该是 0x80，但实物响应 0x88。
接线没错、代码没错，那问题在硬件设计本身。

2. 先澄清 HAL 地址格式（排除这一干扰）

STM32 HAL 的 I2C API 使用8-bit 地址格式：7-bit 硬件地址左移 1 位，留出 R/W 位。这部分不是问题。

A1A07-bit 地址HAL 8-bit 值
GNDGND0x400x80
GNDVS0x410x82
VSGND0x440x88 ← 实测
VSVS0x450x8A

实测 0x88 能工作 ⇒ 芯片实际响应 7-bit 地址 0x44 ⇒ 芯片把 A1 识别成 VS，A0 识别成 GND。

3. 根因：INA226 的 A0/A1 不是普通数字输入

3.1 数据手册怎么说

TI INA226 数据手册 (SBOS547) 8.5.1 节：

"A0 and A1 can be configured to one of four signals (SCL, SDA, GND, or V+), allowing the device to respond to 16 different possible addresses."
"The state of pins A0 and A1 is sampled on every bus communication event."

INA226 的 A0/A1 是 四态采样引脚（quad-state），每个引脚可识别 4 种状态：

识别状态芯片怎么判断推荐外部接法
GND引脚电压 ≈ 0V直接短接到 GND（0Ω）
VS引脚电压 ≈ VS直接短接到 VS（0Ω）
SDA引脚与 SDA 同步跳变直接连到 SDA 走线
SCL引脚与 SCL 同步跳变直接连到 SCL 走线

TI 的 Typical Application 电路图里，A0/A1 是直连，没有任何电阻。

3.2 这块板子的接法

网表 驱动板Netlist_Schematic1_2026-03-25.tel：

'$1N135' ; R41.1 U11.1    ← U11 pin 1 (A1) → R41 → ...
'$1N134' ; R37.1 U11.2    ← U11 pin 2 (A0) → R37 → ...
GND ;  ... R37.2 R41.2 ...  ← R41/R37 另一端都在 GND 网络

电阻值表（line 43）：R37, R41 = 10kΩ。

拓扑示意：

⚠ 问题：设计者按普通数字逻辑输入思路，用 10kΩ 下拉到 GND。但对 INA226 的四态采样电路来说，10kΩ 阻抗太大。

3.3 为什么 10kΩ 下拉会被误识别

INA226 采样 A0/A1 时，引脚内部会注入少量偏置电流来检测外接拓扑：

  • 如果外部直连 GND（0Ω）→ 引脚压降 ≈ 0V → 识别为 GND 态
  • 如果外部直连 VS（0Ω）→ 引脚电压 ≈ VS → 识别为 VS 态
  • 如果外部经 10kΩ 接 GND → 偏置电流 × 10kΩ 产生电压降，具体被识别成什么状态完全由芯片内部偏置设计决定，并且 A0 和 A1 的偏置不一定对称

实测结果表明：这个批次的 INA226 对于 10kΩ 下拉到 GND 的拓扑：

引脚外部接法芯片识别结果
A1 (pin 1)10kΩ → GND识别为 VS 态 ❌（不符合设计意图）
A0 (pin 2)10kΩ → GND识别为 GND 态 ✅

→ 合成地址 = A1=VS, A0=GND = 7-bit 0x44 = HAL 0x88

A0 和 A1 接法完全一样却被识别成不同状态，说明这是不稳定的灰色地带，依赖芯片内部偏置的不对称性。

4. 原理图备注里的线索

驱动板原理图 INA226 附近有两处备注：

① 地址：1000000 — 设计者按 A0=A1=GND 计算的"理论地址"
② 添加3V3焊盘，以供地址可选 — 预留了飞线焊盘

备注 ② 其实间接承认：设计者自己也知道 10kΩ 下拉不保险，才特意留了 3V3 焊盘作为补救。但因为三颗芯片共用 I2C 总线上的多个 INA226 是靠 A0/A1 区分地址的（本项目每路 I2C 只有一颗，所以没有冲突问题），这个 bug 就没有触发诊断。

5. 如何用测试确认这个假设

核心思路：分别做"电压测量"和"扫描 + 修改硬件"两组实验，逐层排除其他可能。

实验 A：静态电压测量（5 分钟，不动硬件）

1
板子上电，让固件正常运行到 INA226 初始化之后（代码用 0x88 能读到的状态）。
2
让 I2C 总线处于空闲（比如在 main.c 初始化 INA226 之后加一个 HAL_Delay(5000)，然后在 ≥1 秒的窗口里测量）。
3
用万用表 DC 电压档，黑表笔接 GND，红表笔分别触到 U11/U12/U13 的 pin 1 (A1) 和 pin 2 (A0)。
4
对照下表判断：
A1 (pin 1) 电压A0 (pin 2) 电压判断
> 2.0 V（接近 3.3V）< 0.3 V✅ 确认是本假设：A1 被识别为 VS，A0 为 GND
介于 0.3V ~ 2.0V任意⚠ 灰色地带，识别不稳定，更坐实"10kΩ 不可靠"
< 0.1V< 0.1V❌ 两个都是 GND，那理论地址应是 0x80，代码用 0x88 能工作就另有原因（需扫描排查）
📌 如果三颗 INA226 的 A1 电压都异常高，就是芯片内部偏置+10kΩ 拓扑的系统性问题；如果只有一两颗异常，可能还有焊接或电阻个体差异。

实验 B：I2C 总线扫描（10 分钟，改固件）

独立确认芯片实际响应哪些地址，排除"代码恰好能访问 0x80 和 0x88 两个"的假象。

1
临时在 main.c 初始化后加扫描代码，扫 0x00 ~ 0xFE（步长 2，仅偶数 = 写地址）： 
printf("I2C1 scan:\r\n");
for (uint8_t addr = 0x02; addr < 0xFE; addr += 2) {
    if (HAL_I2C_IsDeviceReady(&hi2c1, addr, 2, 50) == HAL_OK) {
        printf("  found 0x%02X\r\n", addr);
    }
}
/* I2C2、I2C3 同样做一遍 */
2
预期输出：
I2C1 scan:
  found 0x88         ← 只有 0x88 有响应；如果同时出现 0x80，说明地址解码还有问题

I2C2 scan:
  found 0x88

I2C3 scan:
  found 0x88
3
如果 只有 0x88 响应（没有 0x80、0x82 等），就坐实"A1 被识别为 VS，A0 被识别为 GND"，硬件地址只有 0x44 一个。

实验 C：飞线验证（20 分钟，改硬件，终极排除）

如果前两个实验结果都符合假设，再做这个验证"10kΩ 电阻值"是根因，而不是别的什么。

1
选一颗 INA226（比如 U11），用镊子或飞线把 R41 直接短路（或者把 R41 拆掉，换 0Ω 跳线）—— 这会让 A1 硬拉到 GND。
2
重新上电，先量 A1 (pin 1) 电压：应该 = 0V（被强制 GND）。
3
跑实验 B 的扫描代码。预期结果：
  • ✅ U11 所在的 I2C1 扫描结果变成 0x80（因为 A1 现在真的 GND 了），不再是 0x88
  • U12、U13 保持 0x88 不变
如果观察到这个变化，完全坐实根因是 10kΩ 下拉被误识别。
✅ 实验 C 做完后恢复 R41（把飞线拿掉），避免和其他两颗的地址冲突（本项目不会冲突，因为三颗在不同 I2C 总线上，但保持整齐比较好）。

6. 根据实验结果的处置建议

实验 A 结果 实验 B 结果 处置建议
A1≈3.3V, A0≈0V 只响应 0x88 ✅ 假设成立。保持代码 0x88 不动，在 main.c INA226_Init 处加一行注释说明"实测地址 0x88，原理图 10kΩ 下拉被误识别为 VS 态，下一版改板改 0Ω"。
A1 电压中间值 只响应 0x88 ⚠ 假设成立但更危险。强烈建议下一版改板，把 R41 换 0Ω（或直连 GND），避免不同批次 INA226 识别结果漂移。
A1≈0V, A0≈0V 只响应 0x88 ❓ 假设不成立。电压测到都是 GND 但响应 0x88，可能是假芯片、代码 bug 或其他问题，需进一步排查。
任意 同时响应 0x80 和 0x88（或其他） ❌ 更复杂，可能有地址冲突或芯片异常，需更深入排查。

7. 下一版 PCB 改版建议

方案 1：保持地址 0x88（改动最小）

  • 把 R41 下拉端（R41.2）从 GND 改到 D3V3 → A1 直连 VS
  • R37 保持原状，但改成 0Ω（A0 直连 GND）
  • 代码 0x88 保持不变

方案 2：改成规范的 0x80（推荐）

  • R41、R37 都改成 0Ω（或直接去掉，焊盘短接）
  • A0、A1 都直连 GND → 识别为 GND 态
  • 代码改成 INA226_Init(..., 0x80, ...)
✅ 无论选哪个方案，核心都是去掉 10kΩ 这个灰色地带电阻，让 A0/A1 有确定的状态。

💡 要点回顾

  1. INA226 的 A0/A1 是四态采样引脚，不是普通逻辑输入。
  2. 数据手册要求 A0/A1 直连 GND/VS/SDA/SCL，不要加上下拉电阻。
  3. 本板用 10kΩ 下拉，阻抗太大，芯片把 A1 识别为 VS、A0 识别为 GND，合成地址从 0x80 变成 0x88。
  4. 代码 0x88 能工作是撞对了，不是原理图"对"。
  5. 用实验 A（测电压）+ 实验 B（I2C 扫描）+ 实验 C（飞线短路电阻）三步验证，就能坐实是不是这个问题。
  6. 长期解法：下一版把 R37、R41 改 0Ω，消除 10kΩ 灰色地带。
