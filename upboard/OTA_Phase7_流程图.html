①½ 关键变量字典（看后面流程图前先看这里）

🔑 boot_params_t 结构体（住在 Params A / Params B 两块 16KB flash 里）

字段类型含义 / 谁写 / 谁读
magicu32 固定 0xA5A5A5A5。用来判断这块 flash 真的是 params 不是乱数据（擦除后是 0xFF）。
boot_slotu32 Bootloader 要跳哪个槽。0=Slot A, 1=Slot B。OTA 完成时 App 改成新槽；回滚时 Bootloader 翻过来。
slot_valid[2]u32×2 对应槽是否被 OTA 正式写过。首次裸机烧 Slot A 时是 0，所以 Bootloader 碰到 valid=0 就跳过 CRC 校验（保留"首次无 params 也能跑"的能力）。
slot_size[2]u32×2 该槽镜像有多少字节。Bootloader 做 CRC 校验时必须拿准确的 size，多校少校都会校错。
slot_crc32[2]u32×2 对应 slot_size 字节的 CRC32（IEEE 802.3）。OTA 完成时 App 算好写入；Bootloader 回滚判定时重新算一遍对比。
slot_version[2]u32×2 版本号。目前只"记录"不参与决策，以后可以加"不允许降级"之类的策略。
pending_updateu32 ★最关键的开关 ★。1 = 刚 OTA 完、还在试用期；0 = 稳态（已确认好用）。
• OTA 完成时 App 置 1
• App 连续跑 10 秒后 Ota_MarkBootOk() 清 0
• Bootloader 只在 pending_update==1 时才数 boot_count、校 CRC、做回滚。这是"平时调试不误伤"的关键。
boot_countu32 试用期内 Bootloader 已经启动过该槽几次。每次进 Bootloader 就 ++（前提 pending==1）。
App 稳定运行后 Ota_MarkBootOk() 清 0。
累到 ≥ MAX_BOOT_ATTEMPTS(=3) → 触发回滚。
sequenceu32 A/B 副本仲裁用的序号。每次写 params 时 = max(A.seq, B.seq) + 1。读取时两份都验 CRC，取 seq 更大的那份。用来解决"写 A 写到一半掉电"类问题。
reserved[3]u32×3 12 字节空位。以后加字段不用改结构体大小。
self_crcu32 前面所有字段的 CRC32。读取时对不上 → 这份副本就废，用另一份。
为什么要两个副本（A / B）？写一份时掉电，flash 可能是半新半旧的字节。双副本 + CRC 保证任何时候至少有一份是完整的。
为什么要 sequence？光有 CRC 只能告诉你"这份完整"，不能告诉你"这份最新"。举例：写 B 先成功、写 A 前掉电 → A 是老但完整、B 是新且完整，没 sequence 会错挑 A。

🔑 RTC->BKP0R（备份寄存器，4 字节）

值含义
0xA0A0A0A0 (BOOT_MARK_A)Bootloader 跳 Slot A 之前写的
0xB0B0B0B0 (BOOT_MARK_B)Bootloader 跳 Slot B 之前写的
为什么不直接用 VTOR 判断自己在哪个槽？
Slot A 和 Slot B 的代码 内容完全一样（只是链接地址不同），启动时 SystemInit 会按编译时常量把 VTOR 写成"自己以为的地址"。所以 VTOR 只反映"我以为我在哪"，不是"我实际在哪"。
备份寄存器是 RTC 里的 RAM，软复位不清零（只有掉电且无 VBAT 电池才会清），Bootloader 在这里留个标记给 App 读，最可靠。

🔑 OTA 协议状态机（Core/Src/ota.c 内部静态变量）

变量含义
s_stateOTA_PROTO_IDLE = 空闲；OTA_PROTO_RECEIVING = 在收数据（会让 OtaProto_IsBusy() 返真）
s_target_slotOTA_START 时决定写哪个槽。规则：看 RTC->BKP0R，自己跑在 A 就写 B，反之亦然。
s_total_size本次 OTA 镜像总字节数（OTA_START 时带过来）
s_version本次镜像版本号
s_next_offset期望下一个 DATA 帧的偏移。强制单调递增，防止乱序
s_reset_pendingOTA_END 成功后置 1。主循环看到就 HAL_Delay(50) + NVIC_SystemReset()（先让 ACK 发出去再复位）

🔑 OTA 帧格式（USART2 上跑）

请求: AA | CMD | LEN | PAYLOAD[LEN] | XOR 应答: AA | CMD | 01 | STATUS | XOR （ACK 复用同一 CMD，靠 LEN=1 区分） CMD 常量 | 含义 0xF1 ESP_CMD_OTA_START | PAYLOAD = size(4B LE) | version(2B LE) 0xF2 ESP_CMD_OTA_DATA | PAYLOAD = offset(4B LE) | data(≤128B) 0xF3 ESP_CMD_OTA_END | PAYLOAD = expected_crc32(4B LE) 0xF4 ESP_CMD_OTA_ABORT | PAYLOAD 空 STATUS 常量 (ACK 的 1 字节 payload) | 含义 0x00 OTA_STATUS_OK | 成功 0x01 OTA_STATUS_BAD_STATE | 状态机不对（比如没 START 就 DATA） 0x02 OTA_STATUS_BAD_PARAM | 偏移错 / size 越界 / 未 4 字节对齐 0x03 OTA_STATUS_FLASH_ERR | 擦或写 flash 失败 0x04 OTA_STATUS_BAD_CRC | OTA_END 时 CRC 对不上

为什么 pending_update=0 时不做 CRC 校验？
如果每次启动都校 CRC，那 ST-Link 在线调试让 App 卡死、反复复位时：
→ Slot A 本身好好的，但你在 Bootloader 里 "数圈" 并最终切到了空的 Slot B → 砖！
所以稳态下 Bootloader 什么都不管，只有刚 OTA 完的"试用期"才做监视。

⑤ 复位后的回滚判定

场景 A：新镜像正常

Boot: pending=1, boot_count 0→1 CRC(SlotB) ✔, boot_count(1) < 3 → 继续跳 Slot B，写 params App : 起来 → 主循环 10s → Ota_MarkBootOk() → pending=0, boot_count=0 Boot (下次复位): pending=0 → 无条件跳 B
→ OTA 成功

场景 B：新镜像一起来就炸

复位 1: boot_count 0→1, 跳 B → App 崩/IWDG 复位 2: boot_count 1→2, 跳 B → App 崩/IWDG 复位 3: boot_count 2→3 ≥3 → 翻 boot_slot = A → pending=0, boot_count=0 → LED1 闪 4 次 → 跳 Slot A（旧版、稳定）
→ 自动回滚到 A

场景 C：Slot B 被写坏

ST-Link 硬伤 Slot B 起始 4 字节 复位: pending=1, boot_count 0→1 CRC(SlotB) ✘ → 立即回滚 翻 boot_slot = A, LED 闪 4 次
→ 秒回 A，无需等 3 次

场景 D：平时调试 App 卡死（无 OTA）

params.pending_update = 0 ST-Link 复位 N 次 Boot: pending=0 → 不数圈，不校 CRC 直接跳当前 boot_slot (永远不会切到对面)
→ 不会误伤，无论复位多少次

⑥ 双副本 params 的鲁棒性

写入时： 读取时： seq = max(A.seq, B.seq) + 1 读 A, 读 B (各自验 self_crc) 写 B（先） 两份都有效 → 取 seq 更大的那份 写 A（后） 只一份有效 → 用它 两份都坏 → 返回 false 掉电时机： ① 写 B 之前掉电 → A/B 都是老版本 OK ② 写 B 中掉电 → B 坏 CRC, 用 A（老但完整）OK ③ 写 B 后写 A 前 → B 新 seq, A 老 seq，读到 B，OK ④ 写 A 中掉电 → A 坏 CRC, 用 B（新且完整）OK
任何一步掉电都能恢复到一个"完整、最新可用"的 params 副本。

⑦ 两种看门狗的分工

IWDG（独立看门狗）"软看门狗" = boot_count
保护什么代码真的跑飞/死循环代码能跑，但新镜像有 bug（永远不到 MarkBootOk）
触发时机4s 不喂狗 → 硬件复位3 次复位都没到 10s → Bootloader 切槽
谁能关一旦开启无法关，只能喂Ota_MarkBootOk() 清
谁喂Bootloader 启动，App 主循环喂Bootloader 数，App 清
IWDG 保护 "CPU 还在转" 层面的故障；boot_count 保护 "代码在跑但行为错误" 层面的故障。
两者叠加后：新镜像只要有任何一种挂掉模式，都会在最多 3~4 次复位内自动回滚。

⑧ 关键约束复盘

  • USART2 归 ESP32 → Bootloader 全程静默（只 LED 指示）
  • OTA 共享 UART → App 在 OtaProto_IsBusy() 期间不发 STATUS，防 ACK 污染
  • 调试不能误回滚 → pending_update=0 时 Bootloader 什么都不校、什么都不写
  • 掉电安全 → params 双副本 + sequence 仲裁，任意单点掉电都能恢复
  • 卡死自愈 → IWDG 硬复位 + boot_count 逻辑回滚，两层兜底
